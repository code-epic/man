\chapter{Onboarding e Identidad}
\label{ch:onboarding}

Todo nodo en \sdc{} debe tener una identidad unica para garantizar la auditoria y trazabilidad de todas las operaciones. Este capitulo guia al usuario a traves del proceso de configuracion inicial y vinculacion del dispositivo al ecosistema.

\section{Configuracion de Identidad}
\label{sec:config-identidad}

Al iniciar la aplicacion por primera vez, el sistema presenta una pantalla de bienvenida que solicita la configuracion basica de identidad. Este paso es fundamental para el funcionamiento correcto de todas las caracteristicas de auditoria y seguridad.

\begin{quote}
\textbf{Importancia del Nombre del Equipo}\\
\small
Es vital asignar nombres descriptivos y consistentes con la nomenclatura de tu organizacion. Este nombre aparecera en todos los logs del Monitor de Sistema y sera utilizado para identificar el origen de las operaciones durante auditorias forenses.
\end{quote}

\subsection{Mejores Practicas para la Nomenclatura}

Se recomienda seguir estos patrones para los nombres de equipo:

\begin{itemize}[leftmargin=*]
    \item \textbf{Formato}: \texttt{[ambiente].[usuario]} o \texttt{[ubicacion].[funcion]}
    \item \textbf{Ejemplos validos}:
        \begin{description}
            \item[prod.server01] Servidor de produccion
            \item[dev.carlos] Estacion de desarrollo  
            \item[hq.finanzas] Equipo de finanzas en sede principal
        \end{description}
    \item \textbf{Evitar}: espacios, caracteres especiales o nombres genericos como PC-1 o Equipo
\end{itemize}

\subsection{Generacion del Client ID}

El \keyconcept{Client ID} es un identificador unico generado mediante algoritmos criptograficos que asegura:

\begin{enumerate}[leftmargin=*]
    \item \textbf{Unicidad}: No existen dos dispositivos con el mismo ID en el ecosistema
    \item \textbf{No reversibilidad}: No se puede determinar el hardware original desde el ID
    \item \textbf{Integridad}: Cualquier modificacion del entorno invalida el ID
\end{enumerate}

\begin{quote}
\textbf{Firma Digital de Operaciones}\\
\small
El Client ID se utiliza para firmar digitalmente todas las operaciones realizadas desde el terminal. Esto asegura que cualquier accion (apertura de documento, conexion a servidor, modificacion de configuracion) pueda ser trazada hasta su origen fisico para auditorias futuras.
\end{quote}

\subsection{Proceso de Vinculacion}

El proceso de vinculacion sigue estos pasos:

\begin{enumerate}
    \item \textbf{Deteccion}: SDC escanea automaticamente las interfaces de red y hardware
    \item \textbf{Validacion}: Verifica que el entorno cumple los requisitos minimos de seguridad
    \item \textbf{Generacion}: Crea el Client ID unico vinculado a las caracteristicas del dispositivo
    \item \textbf{Registro}: Almacena de forma segura la huella digital del nodo
    \item \textbf{Confirmacion}: Muestra el resumen de la vinculacion exitosa
\end{enumerate}

\section{Resumen del Proceso de Onboarding}

El proceso de onboarding completo sigue estos pasos secuenciales:

\begin{enumerate}
    \item \textbf{Inicio}: Lanzamiento de la aplicacion SDC
    \item \textbf{Configurar Identidad}: Introducir nombre de equipo y area
    \item \textbf{Detectar Entorno}: Escaneo automatico del sistema
    \item \textbf{Generar Client ID}: Creacion del identificador unico
    \item \textbf{Verificar Integridad}: Comprobacion de modulos de seguridad
    \item \textbf{Dashboard}: Acceso al panel principal de control
\end{enumerate}

\section{Entorno de Criptografia}
\label{sec:criptografia}

El sistema SDC implementa un robusto entorno de criptografia que garantiza la seguridad de todos los datos desde el momento de la configuracion inicial. Este entorno utiliza tecnologias de vanguardia para proteger tanto los datos en reposo como en transito.

\notacientifica{La criptografia implementada en SDC cumple con los estandares internacionales mas rigurosos, incluyendo AES-256-GCM para cifrado simetrico y Argon2id para la proteccion de credenciales.}

\subsection{Device Secret: Clave Unica del Dispositivo}

Durante el proceso de onboarding, SDC genera automaticamente un \keyconcept{Device Secret}, una clave unica de 32 bytes (256 bits) que identifica de manera criptografica al dispositivo.

\notacientifica{El Device Secret es generado utilizando un generador de numeros aleatorios criptograficamente seguro (CSPRNG), garantizando que cada dispositivo tenga una clave unica e impredecible.}

\begin{caja}{Caracteristicas del Device Secret}
\begin{itemize}
\item \textbf{Longitud}: 32 bytes (256 bits)
\item \textbf{Generacion}: CSPRNG del sistema operativo
\item \textbf{Almacenamiento}: En enclave seguro de la base de datos SQLite
\item \textbf{Usage}: Cifrado de datos locales y derivacion de claves
\end{itemize}
\end{caja}

El Device Secret elimina la predecibilidad que existiria al usar solo la direccion MAC del dispositivo, proporcionando un nivel de seguridad significativamente superior.

\subsection{AES-256-GCM: Cifrado de Datos}

SDC utiliza el estandar \keyconcept{AES-256-GCM} (Advanced Encryption Standard con Galois/Counter Mode) para el cifrado de todos los datos almacenados localmente.

\notacientifica{El modo GCM proporciona tanto confidencialidad como integridad en una sola operacion cryptografica, siendo resistente a ataques de manipulacion de ciphertext (bit-flipping).}

\begin{caja}{Especificaciones AES-256-GCM}
\begin{itemize}
\item \textbf{Clave}: 256 bits
\item \textbf{Nonce}: 96 bits (generado unico por cada operacion)
\item \textbf{Autenticacion}: Etiqueta de autenticacion integrada (tag)
\item \textbf{Modo}: Galois/Counter Mode para cifrado autenticado
\end{itemize}
\end{caja}

La combinacion de una clave de 256 bits con un nonce unico garantiza que el mismo texto plano nunca generara el mismo ciphertext, proporcionando confidencialidad perfecta.

\subsection{Argon2id: Proteccion de Credenciales}

Para la proteccion de contrasenas y credenciales, SDC implementa \keyconcept{Argon2id}, el algoritmo ganador de la Password Hashing Competition.

\notacientifica{Argon2id equilibra la resistencia contra ataques de GPU (alto costo de memoria) con resistencia a ataques de canal lateral (ejecucion en tiempo constante), siendo el estandar moderno preferido para el hashing de contrasenas.}

\begin{caja}{Parametros de Argon2id en SDC}
\begin{itemize}
\item \textbf{Memoria}: 64 MB (resistente a ataques GPU)
\item \textbf{Iteraciones}: 3 pasadas
\item \textbf{Paralelismo}: 4 hilos
\item \textbf{Sal}: 128 bits unico por credencial
\end{itemize}
\end{caja}

\notacientifica{Estos parametros fueron seleccionados para proporcionar un equilibrio optimo entre seguridad y rendimiento en dispositivos de escritorio modernos.}

\subsection{TLS 1.3: Comunicaciones Seguras}

Todas las comunicaciones entre SDC y servidores remotos utilizan \keyconcept{TLS 1.3} (Transport Layer Security), el protocolo mas moderno y seguro para comunicaciones cifradas.

\notacientifica{TLS 1.3 reduce significativamente la latencia de conexion mediante el uso de handshake de un solo viaje (1-RTT) y elimina cifrados obsoletos y vulnerables.}

\begin{caja}{Caracteristicas de TLS 1.3 en SDC}
\begin{itemize}
\item \textbf{Handshake}: 1-RTT (vs. 2-RTT en TLS 1.2)
\item \textbf{Cifrados}: Solo suites AEAD modernas
\item \textbf{Forward Secrecy}: Obligatoria (ECDHE)
\item \textbf{Validacion}: Verificacion de certificado del servidor
\end{itemize}
\end{caja}

SDC rechazara cualquier conexion que utilice versiones anteriores de TLS o configuraciones degradadas, garantizando que ninguna comunicacion insecure pueda establecerse.

\subsection{Flujo de Cifrado Completo}

El proceso completo de cifrado en SDC sigue esta secuencia:

\begin{enumerate}
\item \textbf{Datos en Texto Plano}: Los datos ingresan al sistema en su forma original
\item \textbf{Generacion de Clave}: Se deriva una clave unica utilizando HKDF-SHA256
\item \textbf{Aplicar Argon2id}: La contrasena base se procesa con Argon2id para fortalecerla
\item \textbf{Generacion de Nonce}: Se crea un numero unico de 96 bits para esta operacion
\item \textbf{Cifrado AES-256-GCM}: Los datos se cifran con autenticacion integrada
\item \textbf{Almacenamiento}: Los datos cifrados se persisten de forma segura
\end{enumerate}

\notacientifica{Cada paso de este proceso esta diseñado para proporcionar multiples capas de proteccion, de modo que la compromise de una capa no comprometa la seguridad total del sistema.}
